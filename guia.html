Évaluation par les pairs — Guide pour débutantes

Bonjour ! Pas de souci, on va y aller doucement. Imagine que ton prof t'explique l'exercice autour d'un café ☕

De quoi parle l'exercice ?

Tu es dans une classe avec 13 camarades :

Aya Cristian Cyrille David Guillermo Jenifer Joelle Maximilian Mohammed Naomi Natalia Yolanda Yordano

Chacun a fait un travail : un tableau de bord (dashboard) à évaluer.

Le problème : qui évalue qui ?

Tu ne veux pas que tout le monde évalue la même personne, ni que quelqu'un s'évalue lui-même. Tu veux un système équitable et automatique : le programme décide pour toi.

C'est un peu comme un Secret Santa (Amigo invisible) : chacun reçoit une personne à évaluer, sans se choisir soi-même.

Données à avoir prêtes

Liste integrantes — les 13 noms

Aya Cristian Cyrille David Guillermo Jenifer Joelle Maximilian Mohammed Naomi Natalia Yolanda Yordano

Tu peux aussi les écrire ainsi dans ton code :

Aya, Cristian, Cyrille, David, Guillermo, Jenifer, Joelle, Maximilian, Mohammed, Naomi, Natalia, Yolanda, Yordano

Liens repositorio

Idéalement, un objet ou une liste qui relie chaque nom à l'URL de son projet. Si tu ne les as pas encore, tu peux inventer des liens provisoires et les corriger plus tard.

Ce que le programme doit faire, en 3 étapes

Étape 1 — Créer un « modèle » (la classe Evaluadores)

Un modèle, c'est comme une fiche vierge avec 3 cases à remplir :

Case Signification
revisorQui fait l'évaluation (le correcteur)
revisadoQui est évalué
repositorioOù trouver le travail (lien du projet)

Chaque fiche = une paire « évaluateur → personne évaluée ».

Étape 2 — Remplir une liste avec tout le monde

Tu as déjà la liste des integrantes (les 13 noms).

Tu crées un tableau revisiones : une fiche par personne. Au début, tu ne remplis que revisor (le nom de la personne qui va évaluer).

Exemple :

  • Fiche 1 : revisor = Aya, revisado = ?, repositorio = ?
  • Fiche 2 : revisor = Cristian, revisado = ?, repositorio = ?
  • etc.

Étape 3 — Attribuer les paires (la partie « jeu de règles »)

Pour chaque fiche, le programme choisit au hasard qui sera évalué, avec 2 règles :

Règle 1 — Pas d'auto-évaluation
Aya ne peut pas évaluer Aya. Chacun évalue quelqu'un d'autre.

Règle 2 — Chacun est évalué une seule fois
Sur les 13 personnes, chaque nom doit apparaître exactement une fois dans la colonne « revisado ». Personne n'est oublié, personne n'est évalué deux fois.

Ensuite, tu remplis repositorio (le lien du travail de la personne évaluée).

Image concrète

À la fin, tu pourrais avoir quelque chose comme :

Revisor (évalue) Revisado (évalué) Repositorio
AyaDavidlien du projet de David
CristianNaomilien du projet de Naomi
CyrilleAyalien du projet de Aya
………

Chaque ligne = une mission : « Toi, tu évalues cette personne, voici où trouver son travail. »

Pourquoi c'est un exercice de programmation ?

  1. Organiser des infos (classe + attributs)
  2. Travailler avec des listes (tableau d'objets)
  3. Appliquer des règles (pas soi-même, une fois chacun)
  4. Utiliser le hasard (nombre aléatoire pour choisir)

Point important (pour plus tard)

« Choisir au hasard » + « chacun une seule fois » peut être un peu piégeux : le hasard pur ne garantit pas toujours que tout le monde soit couvert. En classe, on te demande surtout de comprendre la logique ; ton prof t'aidera peut-être à affiner l'algorithme.

Structure de chaque fichier JavaScript (toujours la même)

  1. IMPORTATION DE MODULES → import si tu as funciones.js
  2. DÉCLARATION DE VARIABLES → getElementById / querySelector
  3. DÉCLARATION DE FONCTIONS → bibliothèque + parfois funcionMain
  4. EXÉCUTION DU CODE → addEventListener("click", funcionMain)

Bibliothèque vs Application

Couche Ce qu'elle fait
BibliothèqueFonctions qui ne travaillent qu'avec des données. Entrée = paramètres, sortie = résultat. Tu la copies dans d'autres exercices sans la modifier.
ApplicationTout ce qui parle avec la page : lire les inputs, appeler la bibliothèque, afficher les résultats. Change à chaque exercice.

L'écouteur (addEventListener) ne fait qu'une chose : lancer funcionMain quand l'utilisateur clique sur le bouton.

À quoi sert export ?

Dans claseEvaluadores.js ou revisiones.js, tu écris :

export class Evaluadores { ... }
export const integrantes = [...];

Ça veut dire : « Ces éléments peuvent être utilisés depuis un autre fichier ».

Dans app.js, tu les récupères avec import :

import { integrantes, crearRevisiones } from "./revisiones.js";
import { asignarPares } from "./funciones.js";
  • Sans export → l'autre fichier ne les voit pas.
  • Avec export + import → chaque fichier a son rôle, mais ils sont connectés.

Dans le HTML, il faut aussi :

<script type="module" src="js/app.js"></script>

Le type="module" active les modules dans le navigateur.

Est-ce obligatoire d'utiliser export ?

Non. Ça dépend de comment tu organises le code :

Option Comment Avantages / inconvénients
5 fichiers + modules (recommandé) Classe, données et logique séparées avec export / import Comme au cours, clair et réutilisable
1 seul fichier Tout dans app.js, sans export ni import Plus simple au début, mais tout est mélangé
2 fichiers sans modules (ancien style) Deux balises <script>, variables globales Fonctionne, mais moins clair et moins moderne

Comment les fichiers se connectent

claseEvaluadores.js   revisiones.js        funciones.js
       │                     │                     │
       └──── import ─────────┘                     │
                             │                     │
                      new Evaluadores()     asignarPares
                             │                     │
                             └──── import ─────────┤
                                                   │
                                              app.js  ←  index.html
                                         addEventListener("click")
                                                   ↓
                                            affiche le tableau

Dans le HTML, tu ne charges que app.js. C'est lui qui importe depuis revisiones.js et funciones.js.

En résumé

En une phrase : tu construis un petit robot qui distribue les corrections entre camarades, sans qu'on s'évalue soi-même et sans oublier personne.
